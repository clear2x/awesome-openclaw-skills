# 调研备忘

## 这轮仓库扩充的现实判断

如果目标是“明天早上能交付一个明显更完整的开源仓库”，最重要的不是空谈生态，而是把 **本地能核实的样本** 和 **现有路线图** 先整理成中文可读版本。

## 当前最靠谱的资料来源

### 1. 本地已安装 / 已缓存的 skill

这类来源的优点是：

- 能直接读 `SKILL.md`
- 能看到真实 description
- 便于核对依赖、权限和约束
- 最适合写中文导读

本轮重点就来自这些来源：

- `codex_config/vendor_imports/skills-curated-cache.json`
- 本地 curated skill 目录
- 本地 Lark / OpenClaw skill 目录
- 工作区 `skills/` 与 system skills

### 2. 仓库既有路线图

旧版 `docs/100-skills-plan.md` 已经给出了很多名称，这说明仓库之前已经有扩容方向。

问题不在于“有没有名字”，而在于：

- 缺中文解释
- 缺来源拆分
- 缺进度状态
- 缺“已核对 / 待核对”的诚实分层

## 当前生态判断

### OpenClaw 自带 / 本地样本仍然是最高价值来源

就当前可见面来看，最值得抄的是：

- 飞书系列 skill
- docs / file / media 系列 skill
- GitHub / 部署 / 浏览器自动化 skill
- skill-creator / self-improvement 这类 meta skill

因为它们：

- 触发条件更清楚
- 工作流写得更像“能执行的程序说明”
- 更容易看出安全边界

### 公共生态仍值得看，但不该盲信

公共 registry 很有意义，但这类仓库如果想长期有价值，就不能靠“我搜到很多名字”取胜。

更合理的姿势是：

- 先用本地样本建立标准
- 再慢慢吸收外部来源
- 对外部 skill 保持默认审查心态

## 这轮仓库改造的策略

### 优先做对的事

- README 中文化
- 主清单扩到 100 项中文导读
- 给文档补中文版本
- 把“已核对”和“待核对”分开

### 暂时不追求一步到位的事

- 不强行给每个待核对 skill 补外链
- 不对没核实的 skill 做过强推荐
- 不为了凑数去收高风险或语义不明的条目

## 目前结论

这个仓库现在最像一个“中文精选起步仓库”，而不是一个“完整 registry 镜像”。

这反而是好事，因为它的价值会更稳定：

- 帮人快速找方向
- 帮人看懂好 skill 长什么样
- 帮人避开明显不该推荐的模式
