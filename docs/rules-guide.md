# Rules 写作指南

这个文档讲的不是 skill 本身，而是 **rules 应该怎么写**。

在 OpenClaw 语境里，rules 可以出现在很多地方：

- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `TOOLS.md`
- `SKILL.md`
- 路由 / 权限 / 命令配置

一个好的 rule，不是“听起来正确”，而是“真的能约束行为”。

## 1. 先区分规则类型

### 行为规则

控制语气、默认策略与协作方式。

例如：

- 默认中文交流
- 外部动作先确认
- 群聊里不要抢着说话

### 操作规则

控制工作流顺序。

例如：

- 先读本地文档再回答
- 先做只读检查再做修改
- 命令失败时写入 learnings

### 权限与路由规则

控制谁能做什么、消息进哪里。

例如：

- 哪个 bot 走哪个 agent
- 哪些命令需要提权
- 哪些工具只允许在特定上下文使用

## 2. 一个好 rule 的标准

一个好 rule 往往具备这些特征：

- **具体**：不是空话
- **可执行**：模型知道该怎么做
- **可验证**：事后能看出有没有做到
- **值得占上下文**：不是显而易见的废话

坏例子：

- “要安全。”

更好的版本：

- “删除、覆盖、公开发送前先确认。”

坏例子：

- “要有帮助。”

更好的版本：

- “回答 OpenClaw 配置问题前，先查本地文档或已安装 skill。”

## 3. 规则该写在哪

### 写到 workspace 文档里

当它是稳定、长期、跨任务都成立的规则时：

- `AGENTS.md`：流程、边界、习惯
- `SOUL.md`：语气、人格、判断方式
- `USER.md`：用户偏好与协作上下文
- `TOOLS.md`：环境特性与工具坑点

### 写到 skill 里

当它只对某一类任务有效时。

例如：

- 文档编辑 skill 规定“覆盖写入前必须说明风险”
- 安全审计 skill 规定“没有明确要求就不触发安全模式”

### 写到系统配置里

当它需要强约束、不能只靠语言遵守时。

例如：

- 命令授权
- allow / deny list
- 路由与多 agent 绑定

## 4. 写 rule 时最容易犯的错

### 太抽象

“尊重隐私”“谨慎操作”这类话没有错，但往往不够用。

要把它改成真正能执行的句子：

- 不把私有工作区内容带进群聊
- 不在未确认前发送消息到外部平台
- 不运行 destructive command

### 太宽泛

一个 rule 想管所有事，通常最后什么也管不好。

比起：

- “所有任务都要非常认真地检查”

更好的是：

- “做公开发布、权限修改、基础设施变更前必须二次确认”

### 跟系统能力冲突

如果某件事应该用配置硬控，就不要只写成 prose。

例如：

- 高风险命令需要 approval
- 某些工具默认禁用

## 5. 群聊规则尤其重要

很多 agent 最容易翻车的地方就是群聊。

建议把这些写清楚：

- 只有被点名或能明显增值时才发言
- 不要连续刷三条碎消息
- 能用 reaction 就别强行发一段话
- 不要把自己当成用户本人发言

## 6. 文档与记忆规则也很关键

一个强 agent 通常会把这些写死：

- 需要长期记住的事，写进文件
- 每日记忆放 `memory/YYYY-MM-DD.md`
- 长期记忆放 `MEMORY.md`
- 命令失败或被纠正时，写入 learnings

## 7. 一个简单 checklist

加 rule 之前，先问自己：

- 这个 rule 到底防什么故障？
- 它够不够具体？
- 它应该写在 skill、workspace，还是 config？
- 三个月后再看，这句话还成立吗？
- 它会不会和已有 rule 打架？

如果这些问题答不清，它大概率还需要重写。
