# 收录标准与排除标准

这个仓库优先追求三件事：

- **信息密度**：不是堆名字，而是提供值得参考的条目
- **安全边界**：不为高风险自动化洗白
- **真实可用**：最好能追溯到本地 skill、官方样本或清晰来源

## 收录时优先看什么

一个 skill 更容易进入主清单，如果它大致满足这些条件：

### 1. 目的清楚

- 什么时候该触发，读者一看就知道
- 它在解决一个真实存在、会重复出现的问题

### 2. 工作流清楚

- 它不是只有一句“帮我做事”
- 它说明了步骤、工具、限制或前置条件

### 3. 风险可理解

- 会不会写数据、发消息、改权限、改基础设施，要能说明白
- 如果有副作用，是否要求确认、是否有回滚思路、是否先做只读检查

### 4. 文档质量够好

- 描述简洁
- 结构清晰
- 没有明显的 prompt 膨胀
- 有适用场景，也有不适用场景

### 5. 维护性还行

- 依赖能看懂
- 没有明显处于失效状态
- 不是一眼看去就“写完再也没人碰过”的东西

## 我们特别偏爱哪些 skill

- 读多写少的 skill
- 诊断型而不是破坏型的 skill
- 权限与依赖讲得明白的 skill
- 作用域窄、审计成本低的 skill
- 能让后来人学会“怎么写 skill”的 skill

## 什么类型会被谨慎对待

### 1. 天生高风险

例如：

- 秘密提取
- 隐蔽监控
- 伪装成效率工具的越权操作
- 大规模骚扰式外发
- 没有保护栏的 destructive shell automation

### 2. 太模糊，不值得信

例如：

- “什么都能用 bash 做”的大 prompt
- 边界不清的万能 skill
- 安全承诺写得很大，但没有实际机制

### 3. 信息密度太低

例如：

- 主要是泛泛而谈的 AI 建议
- 没有领域知识
- 没有复用价值
- 没有具体工作流

### 4. 误导或过时

例如：

- 安装步骤已经失效
- 依赖根本不存在
- 明明不支持却写成“已经可用”

## 对公共 registry 的态度

不管是 ClawHub 还是其他 registry，都只能算 **发现层**，不是信任背书。

也就是说：

- 在 registry 上能搜到 ≠ 值得推荐
- 有下载量 ≠ 安全或高质量
- 公开上传的第三方 skill 依然需要审查

## 对“部分核对 / 待核对”条目的处理方式

这个仓库允许把一些条目先放进观察池，但前提是：

- 如果已经找到本地同名痕迹、同族上游或运行态证据，可以标成 `部分核对`
- 如果还只有名字和方向判断，就明确标成 `待核对`
- 解释里尽量保守
- 不把它包装成已经确认可用的推荐项

## 一个简单判断题

如果某个 skill 的介绍无法回答下面这些问题，它通常还不够成熟：

- 它到底解决什么问题？
- 什么时候该用，什么时候不该用？
- 它需要什么依赖或凭据？
- 它最大的风险是什么？
- 如果用户是第一次接触这个 skill，能不能读懂？
